SOLID Principles: The Foundation of Maintainable Software Design

Software design isn’t just about writing code ‚Äî it’s about writing clean, extensible, and maintainable code. The SOLID principles, introduced by Robert C. Martin (Uncle Bob), offer a proven foundation for designing high-quality, modular systems.

‚úÖ These five principles promote separation of concerns, reduce code smells, and enhance reusability.

  • S ‚Äî Single Responsibility Principle (SRP)
  • O ‚Äî Open-Closed Principle (OCP)
  • L ‚Äî Liskov Substitution Principle (LSP)
  • I ‚Äî Interface Segregation Principle (ISP)

‚Äã1. Single Responsibility Principle (SRP)

“A class should have only one reason to change.”
The Single Responsibility Principle states that a class should only have one responsibility or function. If a class is handling multiple responsibilities, it becomes harder to maintain and debug.

üî¥ Violating SRP:

class Report {
    public void generateReport() {
        // Logic to generate report
    }
    public void saveToFile(String filename) {
        // Logic to save report to a file
    }
    public void printReport() {
        // Logic to print report
    }
}

üëâ Problem: The Report class is doing too much ‚Äî logic, persistence, and output.

‚Äã‚úÖ Following SRP:

class ReportGenerator {
    public void generateReport() {
        // Logic to generate report
    }
}
class FileSaver {
    public void saveToFile(String filename, ReportGenerator report) {
        // Logic to save report
    }
}
class ReportPrinter {
    public void printReport(ReportGenerator report) {
        // Logic to print report
    }
}

üëâ Solution: Each class handles one responsibility only ‚Äî easier to maintain and extend.

2. Open-Closed Principle (OCP)

“Software entities should be open for extension, but closed for modification.”
You should be able to add new functionality without changing existing, tested code, thus preventing unintended bugs.

üî¥ Violating OCP

class PaymentProcessor {
    public void processPayment(String paymentType) {
        if (paymentType.equals("CreditCard")) {
            // Credit card payment logic
        } else if (paymentType.equals("PayPal")) {
            // PayPal payment logic
        }
    }
}

üëâ Problem: Adding a new method like UPI requires modifying this class.

‚úÖ Following OCP

interface PaymentMethod {
    void pay();
}
class CreditCardPayment implements PaymentMethod {
    public void pay() {
        // Credit card payment logic
    }
}
class PayPalPayment implements PaymentMethod {
    public void pay() {
        // PayPal payment logic
    }
}
class PaymentProcessor {
    public void processPayment(PaymentMethod paymentMethod) {
        paymentMethod.pay();
    }
}

üëâ Solution: New payment types can be added via new classes, no changes to PaymentProcessor.

3. Liskov Substitution Principle (LSP)

“Derived classes must be substitutable for their base classes without affecting the correctness of the program.”
This principle ensures that a subclass should be able to replace its parent class without affecting the application’s behavior.

üî¥ Violating LSP

class Rectangle {
    int width, height;
    public void setWidth(int width) { this.width = width; }
    public void setHeight(int height) { this.height = height; }
}
class Square extends Rectangle {
    public void setWidth(int width) {
        this.width = width;
        this.height = width;
    }
    public void setHeight(int height) {
        this.width = height;
        this.height = height;
    }
}

üëâ Problem: Square violates the expected behavior of Rectangle.

‚úÖ Following LSP

interface Shape {
    int getArea();
}
class Rectangle implements Shape {
    private int width, height;
    public Rectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }
    public int getArea() {
        return width * height;
    }
}
class Square implements Shape {
    private int side;
    public Square(int side) {
        this.side = side;
    }
    public int getArea() {
        return side * side;
    }
}

üëâ Solution: Both shapes now follow expected behavior without breaking substitutability.

4. Interface Segregation Principle (ISP)

“Clients should not be forced to depend on interfaces they do not use.”
Instead of having a large interface with multiple methods, we should split it into smaller, more specific interfaces.

üî¥ Violating ISP

interface Worker {
    void work();
    void eat();
}
class Robot implements Worker {
    public void work() {
        // Robot working
    }
    public void eat() {
        // Robots don't eat!
    }
}

üëâ Problem: Robot is forced to implement eat() unnecessarily.

‚úÖ Following ISP

interface Workable {
    void work();
}
interface Eatable {
    void eat();
}
class Human implements Workable, Eatable {
    public void work() { /* Working */ }
    public void eat() { /* Eating */ }
}
class Robot implements Workable {
    public void work() { /* Robot working */ }
}

üëâ Solution: Clients implement only what they need.

5. Dependency Inversion Principle (DIP)

“Depend on abstractions, not on concrete implementations.”
High-level modules should not depend on low-level modules but on abstractions.

üî¥ Violating DIP

class MySQLDatabase {
    public void connect() {
        // Connection logic
    }
}
class Application {
    private MySQLDatabase database;
    public Application() {
        database = new MySQLDatabase();
    }
}

üëâ Problem: Application is tightly coupled to MySQLDatabase.

‚úÖ Following DIP

interface Database {
    void connect();
}
class MySQLDatabase implements Database {
    public void connect() {
        // Connection logic
    }
}
class Application {
    private Database database;
    public Application(Database database) {
        this.database = database;
    }
}

üëâ Solution: Application can now use any Database implementation (e.g., MongoDB, Oracle, etc.)

Conclusion

  • Modular and reusable
  • Easy to test and debug
