1. Overview

An in-memory cache stores frequently accessed data directly in memory to provide fast reads and reduce expensive operations like database calls or remote API requests.

In this project, you build a cache from scratch with:

  • Bounded capacity
  • Deterministic eviction policies (LRU / LFU)
  • Thread safety
  • Clean, extensible design (policy as a plug-in)

This is not a wrapper around existing libraries — it focuses on internal mechanics and engineering judgment.

2. What You Will Build

Core Capabilities

  • get(key) → return value if present, update usage stats
  • put(key, value) → insert/update, evict when capacity is full
  • Eviction policies:
    • LRU: evict least recently used entry
    • LFU: evict least frequently used entry (tie-breaking required)

Non-Goals (For This Project)

  • Framework-specific integration
  • LeetCode-style problem solving pages
  • Distributed caching (that’s a separate capstone)

3. Algorithms and Data Structures

LRU — Least Recently Used

Idea: evict the entry that hasn’t been accessed for the longest time.

Typical design:

  • HashMap for O(1) lookup: key -> node
  • Doubly Linked List to maintain recency:
    • Move node to head on get/put
    • Evict tail on overflow

Complexity:

  • get() → O(1)
  • put() → O(1)

LFU — Least Frequently Used

Idea: evict the entry with the lowest access frequency.

Typical design needs:

  • key -> (value, freq, node)
  • freq -> list of keys (bucketed by frequency)
  • Track minFreq to evict in O(1)
  • Tie-breaker within same frequency (commonly LRU within bucket)

Complexity (target):

  • get() → O(1)
  • put() → O(1)

4. Engineering and Tech Concepts

This project is about engineering, not libraries.

Concepts You Practice

  • Thread safety: correctness under concurrent access
  • Locking strategy:
    • coarse-grained lock (simple, lower throughput)
    • finer-grained / segmented locks (more complex, better throughput)
  • Interface-driven design:
    • Cache<K,V>
    • EvictionPolicy<K>
  • Separation of concerns:
    • cache storage vs eviction tracking
  • Performance trade-offs:
    • contention vs simplicity
    • memory overhead vs speed

Optional Extensions (Stretch)

  • TTL / Expiration
  • Metrics (hit/miss ratio, eviction count)
  • Read-write locks vs synchronized
  • Background cleanup (for TTL)
  • Size-based eviction (bytes), not only entry count

5. Why This Project Is Great

This is a high-signal project because it:

  • Combines Data Structures + Algorithms + Concurrency
  • Reflects a component used in real production systems
  • Starts simple (LRU) and naturally becomes deep (LFU + concurrency)
  • Helps you practice trade-offs and evolution, not just coding

6. What You Should Be Able to Explain

After completing this project, you should be able to explain:

  • Why HashMap + Doubly Linked List is the standard LRU design
  • Why LFU is harder than LRU (frequency buckets, ties, minFreq)
  • How concurrency changes design (correctness vs throughput)
  • Where contention appears and how you’d improve it
  • How you would extend the cache safely (TTL, metrics, memory limits)

One-Line Takeaway

Small component, huge learning value — caching teaches how real systems behave under pressure.
